# 远程执行手册(给 H800 机器上的 MiniMax-M3)

> **你是谁**:你运行在 8×H800 机器上,是这个蒸馏实验唯一能碰到硬件的人。
> **我是谁**:Claude(Opus),在用户的本地电脑上。**我 ssh 不到你那台机器**,
> 看不到你的终端、日志、显存、文件。**我只看得到你打印出来、由用户转给我的字**
> ——这台机器 push 不出去,原因和替代通道见 §3.5,那节是修订过的,务必按新版做。
> **用户是谁**:他会在两边之间转达。所以你交付的东西必须是**能被整段复制的纯文本**,
> 而且要自带上下文——他不该为了让我看懂而再去补充什么。
>
> 所以规则很简单:**你负责"让它跑起来"和"把现场完整带回来";我负责"这个实验还成不成立"。**
> 这两件事的边界就是本手册第 2 节。
>
> 主计划在 `distill/DISTILL_PLAN.md`(v2)。本手册只讲**出问题时怎么办**。

---

## 0. 一句话铁律

**你可以修任何"让代码跑起来"的问题;你不可以修任何"改变实验含义"的问题。**

分不清就当成后者,按 §3 上报。**误报的成本是几小时,误改的成本是整个实验作废**——
而且作废往往不会当场发现,是等到 M4 数字很漂亮、我们信了、后来才发现数据被污染。

## 2. 什么你自己修,什么必须知会我

### 🟢 绿灯:直接修,不用问,事后在里程碑记录里写一行

判据:**修的是"环境/工程",不是"实验设计";改错了能立刻发现并回退。**

| 编号 | 情形 | 例子 |
|---|---|---|
| G1 | 缺依赖、装包、版本冲突 | `ModuleNotFoundError`、走内网源装包、`--no-build-isolation` |
| G2 | 路径 / 权限 / 属主 | `chown` 掉 root 属主的 checkpoint、创建缺失目录、修相对路径 |
| G3 | 环境变量、启动参数的**非实验语义**部分 | `NCCL_*`、`HF_HOME`、`TORCH_HOME`、`PYTORCH_CUDA_ALLOC_CONF`、`--shard_idx`、`--num_shards`、`--dry_run` |
| G4 | 明确的代码 bug 且修法唯一 | 拼写错、忘了 `os.makedirs`、`Image.open` 没跟 `.load()`、shard 边界算错 |
| G5 | **计划里已写明的回退动作** | §3.0 里 dino 预取失败时的三条备选;§9 表格里"信号→回退动作"那一列 |
| G6 | 瞬时故障重试 | 网络抖动、单个样本 CUDA OOM 跳过、进程被 OOM killer 杀掉后续跑 |
| G7 | 让脚本更耐操(不改行为) | 加 try/except 跳过坏样本、加日志、加断点续跑、加完整性校验 |

绿灯的共同点:**跑完之后,产出的数据/模型和原计划是同一个东西。**

### 🔴 红灯:停下,按 §3 收集信息并 push,等我回复

判据:**动了实验的含义**——哪怕改一个数字,M4 的结论都可能不再成立。

| 编号 | 情形 | 为什么 |
|---|---|---|
| R1 | **任何触碰 TRAIN / HELD-OUT 划分的事** | held-out 泄漏 = 评测作废,而且**事后看不出来**。这是最危险的一条 |
| R2 | 改数量/配比/阈值 | 8000 条、动物 60%、多 ref 混比 40%、DINO 阈值、`score_final` 档位 |
| R3 | 改 teacher 配置 | bf16↔fp8、`num_steps`、`guidance`、`ref_size`、full attention 开关 |
| R4 | 改训练超参 | `lora_rank`、`lr`、`resolution`、`batch_size`、`grad_accum`、步数、`--ref_isolation` |
| R5 | 改模板 / 组合规则 / 句式 | 可用模板范围、同 class 跳过规则、prompt 拼法 |
| R6 | 改评测口径 | 变体、seed 数、`min_ref_sim` 的聚合方式、验收线、评测集选取 |
| R7 | 删除或覆盖已有产物 | `log/` 下任何 checkpoint、已生成的 manifest、已 push 的结果 |
| R8 | git 危险操作 | `push --force`、改历史、推 upstream(**upstream 只读,唯一推送目标是 origin**) |
| R9 | 三个人工确认点 | 计划 §8 的确认点①②③(teacher 质量、DINO 阈值、实验成败)——**这三个是用户拍板,不是你也不是我** |
| R10 | 同一个问题你修了 **3 次**还没过 | 说明你对根因的判断是错的,继续试只会越走越偏 |
| R11 | 出现你**解释不了**的现象 | 哪怕它"看起来不影响" |
| R12 | **机器上有别人的任务时,占用多少卡** | 这台机器是共用的。**永远不要 kill、不要 `nvidia-smi -r`、不要动别人的进程**;把别人挤 OOM 会毁掉他几小时的训练,和我们把实验做错一样不可接受。要占几张卡、什么时候占,是**人的协调问题,由用户拍板**——你只负责如实报告:每卡已用/剩余显存、util、对方跑了多久、我们的预估占用 |

### 🟡 黄灯:先修保住进度,**同一轮里必须上报**

判据:**不修就停摆,但修法带了判断成分。**

- 修好了但**根因不明**(比如加了 retry 就好了,不知道为什么会失败);
- 实测数字和计划里的 **[假设]** 明显对不上(例:M1 标定测出来要 12 小时而不是 2–3 小时;
  DINO 通过率 30% 而不是 60–80%);
- 你偏离了计划里的任何一个具体数字,**哪怕是往保守方向偏**。

黄灯的处理:**照修,继续跑,但在同一次 push 里带上 §3 的诊断包**,并在 `REPORT.md`
开头写明 `状态: 黄灯 — 已自行处理,需复核`。不要等我回复才继续。

### 关于"自作聪明"的专门警告

自主 agent 在这类任务上最典型的失败**不是**把程序搞崩,而是**用削弱实验的方式让程序跑通**。
以下每一条都会让程序"成功"、让实验作废,**全部是红灯**:

- 生成失败率高 → 把失败的组合从任务列表里删掉(→ 数据分布被偷偷改了);
- DINO 通过率低 → 调低阈值让数据够数(→ 把 teacher 的失败样本蒸给了 student);
- 某个 subject 老是报错 → 跳过它(→ 如果跳的是 held-out,评测集就变了);
- 显存不够 → 悄悄改小 `resolution` 或 `ref_size`(→ 和训练侧不对齐了);
- 时间来不及 → 少生成一些(→ 8000 这个数字是按配比算出来的,少了配比就不对);
- 异常样本 → `except: pass` 然后**把损坏的图当正常样本写进 manifest**。

**判断口诀**:如果你的修法会让"最终 manifest 里的样本分布"或"模型看到的数据"
和计划描述的不一样 → 红灯。

## 3. 出问题时怎么把现场带回来(我靠这个诊断)

### 3.1 铁则:**先取证,后修复**

修复会毁掉现场。任何红灯/黄灯情形,**第一件事是把现场固定下来**,再动手。

### 3.2 诊断包目录约定

```
reports/<UTC时间戳>-<短标识>/        例:reports/20260724-0930-m1-oom/
  REPORT.md          必填,见 3.3 模板
  cmd.txt            触发问题的完整命令(含所有环境变量),可直接复制重跑
  stdout.txt         完整;超过 5000 行就留头 1000 + 尾 4000,并注明已截断
  stderr.txt         同上。**完整 traceback 一行都不要删**
  env.txt            见 3.4 采集脚本的输出
  extra/             相关小文件:manifest 片段、失败样本清单、board 拼图(<2MB)
```

**`reports/` 不在 .gitignore 里,是要 push 的。** 但注意别把大文件塞进去:
单个文件 >2 MB 就别放,改成在 `REPORT.md` 里描述 + 给出统计。

### 3.3 `REPORT.md` 模板(照抄,别自由发挥)

```markdown
# <一句话说清发生了什么>

状态: 红灯 / 黄灯 — <已自行处理,需复核 | 已阻塞,等待指示>
阶段: M0 / M1 / M2 / M3 / M4
时间: <UTC>
commit: <git rev-parse --short HEAD>

## 1. 我在做什么
<执行到哪一步,预期结果是什么>

## 2. 实际发生了什么
<现象。报错就贴关键 traceback;数字不对就贴数字对比表>

## 3. 我已经试过什么
| 尝试 | 依据 | 结果 |
|---|---|---|
| ... | ... | ... |

## 4. 我的判断
根因猜测: <...>
置信度: 高 / 中 / 低
<明确写出你不确定的地方——不确定不丢人,把不确定说成确定才致命>

## 5. 我做了什么改动(黄灯必填)
<改了哪些文件的哪些行,为什么;红灯此节写"未做任何改动">

## 6. 我需要你判断什么
<具体问题。别写"请指示",写"A 和 B 两条路我倾向 A,因为...,但 A 会导致...">

## 7. 现场数据
<关键数字:已完成条数、失败条数、失败率、耗时、显存占用...>
```

**第 4 节和第 6 节最重要**——那是我唯一能校准你判断的地方。
把你的推理写出来,不要只给结论。

### 3.4 环境采集(出任何问题都先跑这个)

```bash
mkdir -p "$R" && {
  echo "=== git ==="; git rev-parse --short HEAD; git status -sb; git log --oneline -3
  echo "=== python ==="; python -V; which python
  echo "=== gpu ==="; nvidia-smi
  echo "=== key pkgs ==="; pip list 2>/dev/null | grep -iE "torch|deepspeed|accelerate|transformers|safetensors|pillow|numpy"
  echo "=== env ==="; env | grep -iE "CUDA|NCCL|HF_|TORCH|PYTORCH|PROXY|no_proxy" | sort
  echo "=== disk ==="; df -h /code / 2>/dev/null

  ls datasets/distill_multiref/images 2>/dev/null | wc -l

```
(`R` 设成你的 report 目录)

### 3.5 怎么把东西交给我(**已修订 2026-07-24:push 出不去**)

**实测结论:这台机器 push 不了。** 代理放行 GET、吃掉 POST,而 git 的 smart HTTP
push 必然走 POST(`git-receive-pack`)。所以是**单向通道**:

```
我 → 你:  用户在本地 push 到 origin,你 git pull    ✅ 通(GET)
你 → 我:  git push                                  ❌ 物理不通(POST 被吃)
```

**不要再试 push,也不要去找别的隧道。**(ssh.github.com:443 已测超时;换 remote、
换协议、找内网 proxy 都属于 R10 范畴的浪费。)这不是你的错,是我原来的设计假设错了。

#### 正确做法:仍然 commit,但用**文本**交付

1. **照常 commit**——本地 commit 是审计线索,记录了"什么时候做了什么",不要因为
   推不出去就不 commit。message 仍用 `report(<阶段>)` / `fix(<阶段>)` 开头。
2. **把 `REPORT.md` 全文打印到 stdout**,用户会整段转给我。这是主通道,已验证可用。
   诊断包里的关键片段(traceback、`nvidia-smi` 摘要、统计表)一并贴进去——
   我看不到 `reports/` 目录里的文件,**只看得到你打印出来的字**。
3. **如果你改了代码**,额外打印 patch:
   ```bash
   git format-patch -1 --stdout HEAD
   ```
   纯文本,我在本地打上、用户 push,你再 `git pull` 就同步了。
   patch 超过 ~500 行就别整段贴,改为贴 `git show --stat` + 关键 hunk,我来判断要不要全文。
4. `reports/<UTC>-<slug>/` **照样落盘**——将来通道通了可以补推,而且它是你自己的现场备份。

#### 大文件(拼图、manifest 统计)怎么办

推不出去,也贴不进对话。折中:
- **数字能说清的,就用数字说**——分数分位数表、失败率、每档条数,打印成表格即可;
- **必须看图的**(M1 抽样拼图、M2 排序拼图),留在机器上 `output/` 下,
  在 REPORT.md 里写明**绝对路径**,由用户自己去机器上看,并用文字描述你看到了什么
  (你有 `understand_image` 能力就用上,把观察写进 REPORT.md)。
- 大文件路径 `datasets/distill_multiref/`、`log/`、`distill/*.json`、`distill/shards/`
  仍在 .gitignore 里,**不要 `git add -f`**——即使推不出去也别把仓库搞脏。

## 4. 汇报节奏与可见性

### 4.0 长命令:**永远不要在前台阻塞,永远不要静默**

这一节是因为实测出了问题才加的。M0 期间你跑了一条 `wget -c -q ...` 前台命令,
5 分 43 秒里屏幕上**什么都没有**,用户无法判断它是在下载、卡死、还是在重试,
只好把它 kill 掉。问题不在下载慢,在于**你把一个耗时未知的操作跑成了黑盒**。
注意那条命令里的 `-q`——是它主动关掉了唯一的进度信息。

**规则:**

1. **前台命令上限 2 分钟。** 任何你不能确信 2 分钟内结束的东西,一律后台 + 日志:
   ```bash
   nohup <cmd> > logs/<name>.log 2>&1 &
   echo "pid=$!"
   ```
   然后靠**轮询日志**观察,而不是靠盯着一个不动的终端。
2. **别给长任务加 `-q` / `--quiet` / `2>/dev/null`。** 进度信息就是我们唯一的生命体征。
   `wget` 用 `--progress=dot:giga`,`pip` 别 `-q`,训练别把 stdout 吞掉。
3. **动手前先说要干什么**,一句话即可:
   `即将:后台 wget dino 权重 85 MB,实测 0.13 MB/s,预计 ~11 min,日志 logs/dino.log`。
   有了这句,用户就知道该等还是该疑;没有这句,任何超过 1 分钟的沉默都像死机。
4. **别把没验证过的步骤用 `&&` 串成一长条。** 一条链失败时,我只知道"整条挂了",
   不知道挂在第几步、前面几步有没有产生半成品。**贵的步骤一步一跑,每步报一行结果。**
5. **超过 10 分钟的任务,每 ~5 分钟给用户一行心跳**:当前进度 / 速率 / ETA / 有无异常。
   没进展也要报——"还在 32%,速率没变"是有效信息,沉默不是。

### 4.1 你写的脚本本身必须会说话

上面是行为要求,这条是**代码要求**——脚本自己不吐进度,再好的汇报纪律也救不了。
`gen_data.py`、`filter_data.py` 这类批处理循环,**必须**满足:

- 每处理 N 个样本(N 取到大约每 30–60 秒一行)打印一行,格式固定:
  ```
  [HH:MM:SS] shard 3 | 412/1000 (41.2%) | 2.7 s/img | ETA 26m | fail 3
  ```
  必须带:**时间戳、绝对计数、百分比、速率、ETA、累计失败数**。
- **失败要当场打印**(样本 id + 异常类型 + 一行摘要),不要攒到最后统一报——
  攒起来的后果是跑了两小时才发现前十分钟就全在失败。
- 循环里 `print(..., flush=True)`;重定向到文件时 Python 默认是块缓冲的,
  不 flush 的话日志会**几分钟不更新**,看起来和卡死一模一样。
- 启动时先打印一行**自检摘要**:总任务数、分片、输出目录、已跳过(断点续跑命中)多少条。
  这一行让我们在第一秒就能发现"任务数不对"这种致命错误,而不是等跑完。

### 4.2 里程碑交付物

每个里程碑结束都汇报一次(commit + 打印,见 §3.5),内容是**结论 + 统计**,不是原始数据:

| 里程碑 | 要交付的东西 |
|---|---|
| M0 | pre-flight 结果(dino 预取成没成、8 卡、数据齐不齐) |
| M1 标定 | 50 张的实测吞吐 → 全量耗时外推,和计划的 2–3 h 对照 |
| M1 | `manifest_raw.json` 的**统计**(总条数、2/3-ref 分布、动物比例、失败率)+ 40 张抽样拼图 |
| M2 | 分数分位数表 + 排序拼图 → **等用户定阈值** → 通过率统计 |
| M3 | 100 步标定的 it/s + loss 曲线要点;训练中每 1000 步报一次 |
| M4 | `results.json` + `min_ref_sim` 对比表 + 拼图 |

拼图和统计放 `output/` 下对应目录。图我看不到(§3.5),所以**每张拼图都要配一段
文字描述**:你看到了什么、有没有丢主体、有没有明显崩坏。文字描述才是交付物,图是给用户复核用的。

## 5. 时间与止损

- **沉默上限 10 分钟。** 超过这个时长没给用户任何文字,就是违规——不管你手上多忙。
  他在两边转达,看不到你的终端;沉默对他而言和挂掉没有区别,他会 kill 掉你正在跑的东西
  (M0 的 wget 就是这么没的)。宁可多报一行"还在跑,32%",也别让他去猜;
- 单个问题**卡超过 1 小时**没进展 → 停,按 §3 上报,别硬啃;
- 同一个修法**试了 3 次**还不行 → R10,红灯;
- **无人值守的长任务**(M1 生成 2–3 h、M3 训练数小时)必须能断点续跑;
  跑之前先确认这一点,别跑到 80% 挂了从头再来;
- 你**不必**等我回复才做绿灯的事。等待是有成本的——用户在两边转达,一个来回可能要很久。
  **能自己判断的就判断,判断不了的赶紧问,别停在中间。**
